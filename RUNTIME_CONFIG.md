# Runtime Configuration System

This document explains how GearShare implements secure runtime configuration injection for Firebase credentials and API URLs.

## 🎯 Problem Solved

**Before**: Firebase credentials were passed as Docker build arguments (`ARG`), which:

- ❌ Baked secrets into image layers (visible with `docker history`)
- ❌ Required rebuilding images for credential rotation
- ❌ Made images environment-specific (dev/staging/prod)
- ❌ Exposed credentials in CI/CD logs and Docker registries

**After**: Runtime configuration injection:

- ✅ Secrets injected when container starts (not during build)
- ✅ Same image works in all environments
- ✅ Credentials can be rotated without rebuilding
- ✅ `docker history` shows no secrets

## 🏗️ Architecture

### Production Flow (Docker)

```
Container starts
     ↓
docker-entrypoint.sh runs
     ↓
Reads env vars (VITE_FIREBASE_API_KEY, etc.)
     ↓
Generates /usr/share/nginx/html/config.js
     ↓
window.RUNTIME_CONFIG = { FIREBASE_API_KEY: "...", ... }
     ↓
Nginx serves static files + config.js
     ↓
Browser loads index.html
     ↓
main.tsx calls initializeFirebase()
     ↓
loadRuntimeConfig() fetches /config.js
     ↓
Firebase initialized with runtime credentials
```

### Development Flow (Vite)

```
npm run dev starts Vite
     ↓
runtimeConfigPlugin middleware configured
     ↓
Browser requests /config.js
     ↓
Middleware intercepts request
     ↓
Reads process.env.VITE_FIREBASE_API_KEY, etc.
     ↓
Returns JavaScript: window.RUNTIME_CONFIG = { ... }
     ↓
loadRuntimeConfig() fetches /config.js
     ↓
Firebase initialized with .env values
```

## 🔧 Implementation Details

### 1. docker-entrypoint.sh

Shell script that runs before nginx starts:

```bash
#!/bin/sh
set -e

# Generate runtime config.js from environment variables
cat > /usr/share/nginx/html/config.js << EOF
window.RUNTIME_CONFIG = {
  FIREBASE_API_KEY: "${VITE_FIREBASE_API_KEY:-}",
  FIREBASE_AUTH_DOMAIN: "${VITE_FIREBASE_AUTH_DOMAIN:-}",
  # ... more fields
};
EOF

echo "Runtime configuration generated successfully"
exec "$@"  # Start nginx
```

**Key Points:**

- Runs at container startup (not build time)
- Uses `${VAR:-}` syntax for safe fallback to empty string
- Generates JavaScript that sets `window.RUNTIME_CONFIG`
- Executes nginx after config generation

### 2. runtimeConfig.ts

TypeScript module for loading configuration:

```typescript
export async function loadRuntimeConfig(): Promise<RuntimeConfig> {
  // Production: fetch /config.js generated by docker-entrypoint.sh
  if (import.meta.env.PROD) {
    const response = await fetch("/config.js");
    const scriptContent = await response.text();
    eval(scriptContent); // Sets window.RUNTIME_CONFIG
    return window.RUNTIME_CONFIG;
  }

  // Development: fallback to .env values
  return {
    FIREBASE_API_KEY: import.meta.env.VITE_FIREBASE_API_KEY || "",
    // ... more fields
  };
}
```

**Key Points:**

- Async function (returns Promise)
- Production: fetches and evals `/config.js`
- Development: reads from `import.meta.env` (Vite's env vars)
- Cached after first load (singleton pattern)

### 3. firebase.ts

Async Firebase initialization:

```typescript
export async function initializeFirebase(): Promise<void> {
  const config = await loadRuntimeConfig();

  // Configure API client with runtime URL
  configureApiClient(config.API_URL);

  // Check if config is present
  if (!config.FIREBASE_API_KEY) {
    console.error("Firebase configuration missing!");
    return; // Don't crash, just disable Firebase
  }

  // Initialize Firebase
  app = initializeApp({
    apiKey: config.FIREBASE_API_KEY,
    // ... more fields
  });
}
```

**Key Points:**

- Loads runtime config first
- Configures API client before Firebase
- Graceful handling of missing credentials
- Returns early if no API key (doesn't crash)

### 4. main.tsx

App entry point with async initialization:

```typescript
async function init() {
  try {
    await initializeFirebase();
    root.render(<App />);
  } catch (error) {
    console.error("Failed to initialize app:", error);
    root.render(<ErrorComponent error={error} />);
  }
}

init();
```

**Key Points:**

- Calls `initializeFirebase()` before rendering App
- Handles initialization errors gracefully
- Shows error UI if config fails to load

### 5. Vite Middleware (Development)

Plugin that serves `/config.js` in dev mode:

```typescript
function runtimeConfigPlugin(): Plugin {
  return {
    name: "runtime-config",
    configureServer(server) {
      server.middlewares.use((req, res, next) => {
        if (req.url === "/config.js") {
          const config = `
            window.RUNTIME_CONFIG = {
              FIREBASE_API_KEY: "${process.env.VITE_FIREBASE_API_KEY || ""}",
              // ... more fields
            };
          `;
          res.setHeader("Content-Type", "application/javascript");
          res.end(config);
          return;
        }
        next();
      });
    },
  };
}
```

**Key Points:**

- Intercepts `/config.js` requests in development
- Reads from `process.env` (from `.env` file)
- Returns JavaScript that mimics production config
- Allows testing production config pattern locally

## 🚀 Deployment

### Docker Compose

```yaml
services:
  frontend:
    build: ./frontend
    environment:
      # Runtime injection (not build args!)
      - VITE_FIREBASE_API_KEY=${VITE_FIREBASE_API_KEY}
      - VITE_FIREBASE_AUTH_DOMAIN=${VITE_FIREBASE_AUTH_DOMAIN}
      # ... more env vars
    ports:
      - "80:80"
```

**Key Points:**

- No `args` in build section
- All credentials in `environment` section
- Loaded from `.env` file (not committed to git)

### Credential Rotation

To update credentials without rebuilding:

```bash
# 1. Update .env file with new credentials
vim .env

# 2. Restart container (no rebuild!)
docker-compose restart frontend

# Config regenerated with new credentials on startup
```

## 🛡️ Security Benefits

1. **No Secrets in Layers**: `docker history` shows no credentials
2. **Registry Safe**: Images can be pushed to Docker Hub/ECR safely
3. **CI/CD Safe**: Build pipeline doesn't need production credentials
4. **Environment Isolation**: Dev/staging/prod use same image
5. **Rotation Friendly**: Update secrets without rebuilding
6. **Service Worker Access**: Config available to SW via `/config.js`

## 🤝 Contributing

When modifying the runtime config system:

1. **Add new config field**:
   - Update `RuntimeConfig` interface in `runtimeConfig.ts`
   - Add to `docker-entrypoint.sh` generation
   - Add to Vite middleware in `vite.config.ts`
   - Add to `docker-compose.yml` environment section

2. **Test changes**:
   - Build Docker image and verify no secrets in layers
   - Test dev mode with `npm run dev`
   - Test production mode with Docker
   - Run test suite: `npm test`

3. **Update documentation**:
   - Add to `.env.example` if user-facing
   - Update this document if architectural change
